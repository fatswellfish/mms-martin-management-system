# MMS 综合管理系统 - 系统设计文档

## 1. 概述
MMS (Martin Management System) 是一个基于 ISO9001 质量管理体系的综合性组织管理系统。本设计文档详细阐述了系统的整体架构、核心模块的功能以及模块间的依赖关系。

## 3. 核心模块详解

### 3.1 System Core (核心系统)
- **定位**: 提供所有模块共享的基础能力，是整个系统的基石。
- **关键组件**:
  - `models.py`: 定义用户、权限等通用数据模型。
  - `schemas.py`: 定义标准化的 API 请求/响应格式（DTO）。
  - `repository.py`: 提供统一的数据访问接口，支持多种数据库后端。
  - `service.py`: 封装业务逻辑，是各模块调用的核心服务。
  - `router.py`: 提供全局路由，管理 `/system-core` 前缀下的所有端点。

### 3.2 fieldops (组织与规划)
- **功能**: 管理农场的物理结构（农场 -> 猪舍 -> 猪栏）和生产批次的逻辑关系。
- **关键特性**:
  - 实现完整的层级树状数据结构。
  - 提供 `GET /fieldops/tree` 接口，返回可直接用于前端渲染的嵌套对象。
  - 通过 `GET /fieldops/barn/{barn_id}/pens` 获取特定猪舍的所有猪栏信息。
- **技术实现**:
  - 使用 SQLAlchemy ORM 建立数据库关系。
  - 通过 `main_router.py` 和 `location_router.py` 分别处理顶层和具体资源的路由。
- **部署策略**: 作为独立子模块，将使用一个独立的、专用的 PostgreSQL 数据库，不与主项目 `mms` 共享数据库，以便未来可以将其作为一个独立产品进行发布和维护。

### 3.3 Event Engine (事件引擎)
- **定位**: 作为跨多个业务模块的通用事件处理中心，是系统的“神经系统”。
- **核心职责**:
  - 接收来自任何模块的事件（如"健康检查完成"、"饲料到货"）。
  - 对事件进行统一的分类、记录和状态管理（待办、已完成、已取消）。
  - 触发后续的自动化流程或通知。
  - 提供事件历史查询和统计分析功能。
- **技术实现**:
  - 位于 `domain_operations/farm_domain/event_engine/` 目录下。
  - 使用 FastAPI 构建独立的微服务，拥有自己的 `router`, `models`, `schemas`, `service` 层。
  - 包含完整的前端页面（列表、详情、创建），使用 Jinja2 模板引擎。
  - 通过 `GET /api/related-data` 接口动态加载关联的畜舍和批次数据，提升用户体验。

## 4. 模块间依赖关系
- `Q1, Q2, Q3, Q4` 模块均依赖于 `System Core` 模块提供的基础服务。
- `Q1 (fieldops)` 模块为 `Q2, Q3, Q4` 模块提供基础的结构化数据（如猪舍、批次信息）。
- `Event Engine` 模块是所有其他模块的“消息中心”，任何模块都可以向其发送事件。

## 5. 技术栈总结
- **后端框架**: FastAPI
- **数据库**: PostgreSQL (对于 `fieldops` 模块，将使用独立的数据库实例)
- **ORM**: SQLAlchemy
- **前端**: HTML/CSS/JavaScript + Jinja2 模板引擎
- **部署**: Docker + Nginx

## 6. 未来扩展性
- 该架构设计具有极高的可扩展性。未来可以轻松添加新的业务模块（如 `Q5: Sustainability`）或新的事件类型。
- `Event Engine` 可以被扩展为一个强大的工作流引擎，支持复杂的条件判断和并行任务。
- `fieldops` 模块因其独立的数据库和代码结构，具备了作为独立产品的潜力，可以随时进行拆分和商业化。
