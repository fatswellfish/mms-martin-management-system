# MMS 项目执行约束条件 (基于 qwen-flash 模型)

## 核心目标
确保所有任务在 `qwen-flash` 模型（16000 tokens 上下文窗口）的限制下，能够安全、高效地完成，防止上下文溢出导致的性能下降或任务失败。

## 关键约束与策略

### 1. 任务拆分原则 (强制)
- **最大步数**: 任何复杂任务必须被拆分为 ≤3 个独立步骤。
- **每步思考量**: 每一步的内部推理和分析文字长度必须 < 1000 tokens。

### 2. 输出优先级 (强制)
- **优先输出代码/文件列表**: 在每个步骤中，**首要任务是输出具体可执行的代码片段、文件名列表或操作指令**。分析性文字应作为补充说明，且长度严格控制在 <200字。
- **示例**: 
  - ✅ 正确: `已生成文件: mms/modules/fieldops/models/location.py, mms/modules/fieldops/schemas/location.py`
  - ❌ 错误: 大段解释“为什么需要这两个文件”

### 3. 进展汇报机制 (强制)
- **超时汇报**: 如果一个步骤的处理时间超过 30 秒，必须立即主动汇报进展，例如：
  > `【进展汇报】正在处理步骤2，已完成模型类定义，正在进行数据验证逻辑编写。`
- **目的**: 避免用户等待无响应，及时确认系统状态。

### 4. 上下文管理建议 (推荐)
- **使用 `tasks/` 和 `docs/` 分离**: 如前所述，将核心决策文档（`plan.md`, `task1.md`）存于 `docs/`，将当前待执行的具体任务（如 `step1.md`）存于 `tasks/`，可以有效隔离长文本，减少单次请求的上下文负担。
- **避免一次性读取过多文件**: 在执行任务时，仅加载当前步骤所需的文件，避免因读取无关文件而占用大量上下文空间。

## 总结
通过严格执行以上四条约束，可以确保我们与 `qwen-flash` 模型的交互始终处于其能力范围内，实现稳定、高效的开发协作。请将此文档视为本次会话的“技术宪章”。
